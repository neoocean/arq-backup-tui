# Multilingual / Unicode / emoji / long-path handling guarantees

This document defines the contract for how this project preserves and
restores the wide variety of strings that can appear in **file names
and path names**. Every item here is verified by automated tests
(`tests/test_unicode_path_stress.py`, `tests/fixtures_unicode.py`).

## 1. Core guarantees

### 1.1 Byte-level round-trip

When a file from a source filesystem is backed up and then restored,
the following two values are **byte-for-byte identical**:

| Compared values | Guaranteed |
|----------|------|
| `src.read_bytes()` ↔ `restored.read_bytes()` | ✅ |
| The sequence of component names (UTF-8 bytes) used to reach the file in the source tree | ✅ |

This guarantee is independent of which characters are involved
(language / script / normalization form / emoji / ZWJ sequences /
combining characters).

### 1.2 Scope

| Location | Handling |
|------|----------|
| Tree blob child name | `[String]` format: `[is_not_null:1B] + [length:UInt64 BE] + UTF-8 bytes` (exactly per the spec convention) |
| `backupplan.json` / `backupfolder.json` / `backupconfig.json` / `backupfolders.json` | `json.dumps(..., ensure_ascii=False).encode("utf-8")` — non-ASCII characters are stored as native UTF-8 instead of being escaped as `\uXXXX` |
| `backuprecord` (binary plist) | `plistlib` automatically handles them as UTF-8 strings |
| Symlink target | Stored as plaintext via `os.readlink(src).encode("utf-8")` → restored via `os.symlink(target_str, dest)` |
| TUI display labels | Textual + Rich render BMP-external characters (such as emoji) accurately |

### 1.3 Normalization policy: **none**

This library performs **no normalization at all** — neither NFC nor NFD nor NFKC nor NFKD.

- If the source has NFC `한글` (precomposed), the destination keeps it
  as precomposed bytes.
- If the source has NFD `한+ㄱ` (decomposed jamo), the destination
  keeps it as decomposed bytes.
- Combining characters / variation selectors / ZWJ sequences are all preserved.

**Note**: macOS HFS+ (legacy) forces NFD normalization at the
filesystem layer. In that case we receive (and store as-is) what the
source filesystem already converted to NFD — our code behaves
correctly, but the visual notion of "the same character" may not be
preserved. APFS (the modern macOS default) and ext4 / btrfs / xfs
(Linux) are byte-preserving, so they round-trip exactly.

### 2.4 Normalization forms

NFC and NFD are both stored and restored as raw bytes, and they can
coexist in the same directory as long as the host filesystem does not
collapse them together:

- `español` (NFC: `e + s + p + a + ñ(precomposed)`)
- `español_nfd` (NFD: `e + s + p + a + ñ(n + combining tilde)`)
- `한글` (Hangul precomposed)
- `한글_jamo` (Hangul jamo decomposed)

## 3. OS / filesystem-limit handling

### 3.1 Behavior at the limits

This library does **not hardcode** any OS / filesystem limit
(NAME_MAX, PATH_MAX, permissions). The host OS / filesystem enforces
the limit, and when Python's `mkdir` / `write_bytes` calls fail with
`OSError` our code catches and emits an event:

| Failure site | Event emitted | Behavior |
|----------|----------|------|
| `src.iterdir()` permission denied | `dir_read_error` | The directory is treated as having empty children; the backup continues |
| `src.read_bytes()` permission denied | `file_read_error` | The file is treated as 0 bytes (the FileNode is still emitted) |
| `out_path.write_bytes()` disk full | `OSError` propagated as-is | Restore fails |
| Source name is too long | OS could not create it in the first place → walk does not see it | N/A (it is not in the source, so it is not backed up) |

In other words, we operate on the assumption that **"impossible inputs
do not occur"**. Names the filesystem already rejected never reach our
layer, and read failures on individual entries do not abort the whole
backup — we continue best-effort.

### 3.2 OS-specific known limits

| OS / filesystem | Limit | This library's behavior |
|----------------|------|-------------------|
| Linux ext4 | NAME_MAX 255 bytes, PATH_MAX 4096 | Works within the limit |
| Linux btrfs | NAME_MAX 255 bytes | Same |
| macOS APFS | NAME_MAX 255 bytes, PATH_MAX 1024 (default) | Same; aware of the PATH_MAX limit |
| macOS HFS+ | NFD normalization forced, `:` forbidden | Input arrives already in NFD bytes → stored as-is |
| Windows NTFS | Some characters (`<>:"/\\|?*`) forbidden, reserved names (`con`/`nul`/`aux`) | This project is POSIX-first; on Windows the source does not contain those characters |
| SFTP server (most) | Follows the server-side filesystem limits | Backend layer surfaces server OSErrors as RuntimeError |

## 4. Implementation key points

### 4.1 JSON sidecar `ensure_ascii=False`

All four JSON-sidecar emit sites in `arq_writer/backup.py` use
`ensure_ascii=False`:

```python
self.backend.write_all(
    self._cu_path("backupplan.json"),
    json.dumps(plan, indent=2, ensure_ascii=False).encode("utf-8"),
)
```

As a result:

- **Before (bug)**: escape sequences such as `"name": "\\ud55c\\uad6d\\uc5b4-\\ud3f4\\ub354"` appeared
- **Now**: stored verbatim as `"name": "한국어-폴더"`

Non-ASCII folder names round-trip as native UTF-8 bytes throughout
plan / config / folder JSON.

### 4.2 Tree binary [String] encoding

`arq_writer/serialize.py:write_string`:

```python
def write_string(value: Optional[str]) -> bytes:
    if value is None:
        return b"\x00"
    encoded = value.encode("utf-8")
    return b"\x01" + struct.pack(">Q", len(encoded)) + encoded
```

The length is 8-byte big-endian and the payload is UTF-8 bytes. The
maximum length is `2^64 - 1` (effectively unbounded). Any codepoint
passes through unchanged.

### 4.3 Symlink target string

writer (`arq_writer/backup.py:_walk_file`):

```python
if src.is_symlink():
    data = os.readlink(src).encode("utf-8")
    st = src.lstat()  # metadata of the symlink itself
```

restorer (`arq_reader/restore.py:_restore_file_node`):

```python
if stat.S_ISLNK(int(node.mac_st_mode)):
    target = body.decode("utf-8", errors="replace")
    os.symlink(target, out_path)
```

`errors="replace"` is a safe fallback for a corrupt backup. For a
healthy backup the target is restored as raw UTF-8.

### 4.4 Path filter (selective restore) byte-level matching

`arq_reader/restore.py:_PathFilter`:

```python
def matches(self, rel_path: str) -> bool:
    rel_path = rel_path.strip("/")
    if rel_path in self.keep:
        return True
    for k in self.keep:
        if rel_path.startswith(k + "/"):
            return True
    return False
```

String comparison (Python's internal codepoint-sequence comparison) →
matches one UTF-8 byte unit at a time. `paths=["한글폴더"]` matches
`한글폴더/메모.txt`.

## 6. Limitations and future improvements

- **Invalid UTF-8 byte sequences**: Linux can use `surrogateescape` to
  handle ext-encoded non-UTF-8 file names, but this library's
  `write_string` rejects such strings with `UnicodeEncodeError`. In
  practice every modern Linux/macOS filesystem mandates or recommends
  UTF-8, so this rarely happens.
- **Windows long-path mode (`\\?\` prefix)**: this project is
  POSIX-first; Windows native long-path support can be added later if
  needed.
- **Visually identical but distinct codepoints**: cases such as `'A'`
  (U+0041) vs `'А'` (Cyrillic A, U+0410) — visually identical but
  with different codepoints. The library handles raw bytes, so the
  two are recognized as different files (Arq.app behaves the same).
